09 — METHOD

How the register was formulated

Distinct key per source

Each source deduplicated on its natural key — demand number for D2D, use-case id for Al Hasbah, agent id for Agents, title for Deployment and Retreat.

Mapped to the 61-column master schema

Source fields mapped column-by-column to the master layout (e.g. use-case id → aum_demandno, agent_id → aum_agentid, dates, division, status, savings). Unmapped columns left empty rather than filled with placeholders.

State of AI carried from master

The 33 State-of-AI records have no external source file, so they are carried through from the existing master untouched.

Counts reconciled to targets

Per-source distinct counts validated against the requested targets; every count traces back to a raw file.

10 — DATA INTEGRITY

Open items & honest caveats

Flagged rather than papered over, so the headline figures stay defensible under scrutiny.

Retreat +7
AI Retreat is 193, not 186. The raw files hold 193 distinct use cases with no overlap. Either 7 records are out of scope, or the 186 target is stale — pending your decision.
State of AI
No source file to verify. The 33 records match the master but cannot be reconciled to a raw export.
Cost outlier
One 880M cost-saving value excluded from the AED 273.7M headline pending validation; it is ~3× the entire rest of the portfolio combined.
FTE field
FTE-savings column is mixed. State-of-AI rows carry option-set codes, not FTE numbers, so only the clean 80,920 annual hours is reported — a raw sum of the column is not meaningful.
